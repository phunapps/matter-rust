# M7.5 verification report — operational cluster control on a real device

**Status:** COMPLETE — PASSED 2026-06-09 on a Tapo P110M (production
attestation chain). The `control_onoff` example commissioned the device and
then read/toggled `OnOff` and wrote/read `BasicInformation.NodeLabel` over a
fresh operational CASE session, driving the generated `matter-clusters` codecs.

Discharges the M7 validation requirement (CLAUDE.md): "Read and write
attributes on a real device. Invoke commands."

This report records the conformance evidence — against the Matter spec, the
matter.js reference, and the C++ `connectedhomeip` reference (at runtime, via
the device) — and is honest about the limits of that evidence.

## What ran

`cargo run --example control_onoff --features driver,tracing,wiretrace -- \
  --manual <code> --addr 192.168.1.248:5540 \
  --paa-dir <chip>/credentials/production/paa-root-certs \
  --cd-root <chip>/credentials/production/cd-certs --trace-out runs/rust-onoff.jsonl`

Observed (decoded values cross-checked against the captured wire payloads):

| step | operation | wire | result |
|---|---|---|---|
| 1 | read `OnOff.OnOff` (ep1, 0x0006/0x0000) | IM ReadRequest 0x02 → ReportData 0x05 | `true` |
| 2 | invoke `OnOff.Toggle` (ep1, cmd 0x02) | IM InvokeRequest 0x08 → InvokeResponse 0x09 | OK |
| 3 | read `OnOff.OnOff` again | 0x02 → 0x05 | `false` (flipped) |
| 4 | write `BasicInformation.NodeLabel` (ep0, 0x0028/0x0005) | IM **WriteRequest 0x06** → WriteResponse 0x07 | status OK |
| 5 | read `NodeLabel` | 0x02 → 0x05 | `"matter-rust"` (echoed) |

The device physically toggled and persisted the label. The operational TLV on
the wire was confirmed to be the exact output of the `matter-clusters` codecs
(e.g. the `NodeLabel` write payload `…2c02 0b 6d61747465722d72757374` =
context-tag-2 UTF-8 string "matter-rust").

## Conformance evidence

Two independent, conformant references corroborate our bytes for the surface we
implement.

### 1. The C++ reference (`connectedhomeip`) — at runtime, via the device

The Tapo P110M runs a connectedhomeip-derived stack, so device acceptance *is*
a comparison against the C++ reference. It cryptographically validated and
accepted every message:

- **PASE / SPAKE2+** (§3.10) completed — shared secret agreed.
- **Attestation** (§6.2): the device returned DAC/PAI/CD to our challenge; we
  validated the chain against the **production** Tapo PAA + CSA CD roots, and
  correctly *rejected* an earlier run that used CSA *test* roots (negative
  control — wrong-root rejection is itself spec-correct behavior).
- **NOC issuance** (§6.5): the device accepted `AddTrustedRootCertificate` and
  `AddNOC`, i.e. it parsed and validated our Matter-cert encoding + NOC chain.
- **CASE / SIGMA** (§4.13–4.15) completed — the device validated our NOC chain,
  SIGMA transcript hash, destination-id HMAC, and **operational IPK
  derivation**. This is the deepest signal: one wrong byte in the transcript or
  the IPK and the device silently drops Sigma1. It didn't, twice (commissioning
  CASE + a second, fresh operational CASE).
- **Operational IM**: the device's IM engine processed our concrete-path
  Read / Invoke / **Write** and produced correct cluster behavior. This is the
  operational-conformance proof against the C++ reference — stronger than a wire
  diff, because the device is the spec arbiter and acts on our exact bytes.

### 2. matter.js — byte parity (commissioning)

The commissioning dialogue is **10 MATCH / 16 MATCH\* / 0 DIVERGENT** byte-for-
byte against matter.js 0.17.1 (`cargo xtask trace-diff`; see
[`m6-cross-verification.md`](m6-cross-verification.md) for method/semantics).
The `MATCH*` deltas are random material (nonces, ephemeral ECDH keys,
signatures); `THEIRS-ONLY` are matter.js's spec-*optional* extra model-building
reads; `OURS-ONLY` are all allowlisted with rationale (0 unclassified). DAC/PAI
fetch order follows the **C++ reference** (PAI-first) per the repo convention
that connectedhomeip is authoritative for protocol bytes.

### 3. Per-layer test vectors

Each layer additionally has captured-vector unit tests: TLV, Matter certs,
PASE, CASE, IM framing (`matter-interaction` `im_byte_parity`), and the cluster
codecs (`matter-clusters` `byte_parity` vs matter.js 0.16.11 TLV combinators,
M7.4b). The operational payloads observed on the wire above match these
vectors.

## Why there is no operational *trace-diff* number

The original M7.5 aspiration was an operational `trace-diff → 0 DIVERGENT`. That
turns out not to be a clean or meaningful target, for two reasons:

1. **`trace-diff` intentionally cuts the comparison window at
   `CommissioningComplete`** (M6 design) — the operational tail is ignored.
2. **The two controllers structure operational traffic differently.** Our
   example issues a tight 5-message sequence with an explicit `WriteRequest`
   (0x06). matter.js's high-level controller does batched multi-attribute
   structure-discovery reads across **multiple** CASE sessions and emits **no
   `WriteRequest` at all** for the label (it writes via a different internal
   path; confirmed: 0 × opcode 0x06 in its trace). There is simply no
   comparable matter.js operational sequence to byte-diff against.

So the operational layer is verified by **(a)** the C++-reference device
accepting and acting on our exact bytes, and **(b)** codec + IM-framing
byte-parity at the unit level — not by an end-to-end operational wire diff.
`trace-diff` gained IM Write-opcode decoding (0x06/0x07) in M7.5 regardless, so
write messages align on their `(cluster, attribute)` target whenever they fall
inside a compared window.

## A real bug this surfaced

Device validation caught a genuine defect (fixed + unit-tested in this
milestone): after `commission()` hands off to a fresh operational
`SessionManager`, the device keeps MRP-retransmitting its final
commissioning-session frame, addressed to the now-unknown old session id. The
secured-exchange recv loop propagated the resulting `UnknownSession` error and
aborted the first operational read. Fix: the loop now skips inbound frames it
cannot attribute (`UnknownSession`) or decrypt (`DecryptionFailed`), mirroring
its existing unsecured-straggler handling
(`secured_round_trip_skips_unknown_session_frames`).

## Honest limits of the claim

- **Subset, by design.** The IM layer is concrete-path, one-command-per-invoke,
  no subscriptions / events / timed actions / chunked writes / wildcards (M8).
  No multi-fabric management, groups, OTA, or Thread/BLE commissioning yet. The
  *implemented* surface is conformant; the rest is not-yet-built, not wrong.
- **No formal CSA certification.** Evidence is byte-parity vs matter.js +
  live acceptance by a connectedhomeip-derived device + spec/vendor test
  vectors — strong, but not the CSA Test Harness / `chip-tool` conformance
  suite.
- **One device family.** Tapo P110M only (chip-derived). See
  [`tested-devices.md`](tested-devices.md); coverage widens over time.

## Bottom line

For the protocol surface implemented through M7 — TLV, Matter certs,
PASE/SPAKE2+, CASE/SIGMA, secured framing + MRP, commissioning, IM
read/invoke/write framing, and the 10 clusters — `matter-rust` is on-spec,
corroborated independently by matter.js (bytes) and a live C++-reference device
(cryptographic acceptance + correct cluster behavior). The crypto-heavy paths
are the most strongly validated, because a real device fails them silently if
they are off. The open questions are *coverage* (we are a subset) and *formal
certification* (not yet attempted), not correctness of what exists.
