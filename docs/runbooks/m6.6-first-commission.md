# Runbook: first IP commission with `commission_ip` (M6.6)

This is an **operator-touch** procedure, not a CI gate. It commissions a real,
IP-reachable Matter device end to end with the pure-Rust `commission_ip` example,
and cross-verifies the on-wire trace against matter.js.

> A factory-fresh Matter device is **BLE-only** until it has network credentials,
> and BLE commissioning transport is deferred post-v1.0. So the IP path below uses
> a device that is **already on the network** and put into a commissioning window.
> The factory-fresh **Tuya-plug-over-BLE** run named in CLAUDE.md's M6 validation
> remains **deferred until BLE lands** — it is not dropped, just not yet runnable.

## Path A — real LAN device via open commissioning window (production)

1. **Open a commissioning window.** Pick a device already commissioned to another
   ecosystem (Apple Home / Google Home / SmartThings). In that app, choose
   "pair to another app" / "add to another service". It opens a commissioning
   window (Enhanced Commissioning Method) and shows an **11-digit manual code** or
   a QR code. Note: this is a **second-fabric** commission — our controller adds its
   own fabric to the device. Ongoing multi-admin *management* (cross-fabric ACLs,
   fabric removal) stays deferred per CLAUDE.md; only the commissioning act is in scope.
2. **Get production attestation roots.** Clone `project-chip/connectedhomeip` (or
   sparse-checkout `credentials/production/`):
   - `--paa-dir` → `credentials/production/paa-root-certs/` (the example loads every
     `*.der` there).
   - `--cd-root` → the `credentials/production/cd-certs/` directory itself. The
     example loads every `*.der` CD signing certificate there (there are several
     distinct CSA CD signing keys; a device's CD may be signed by any of them, so
     load them all rather than picking one).
3. **Commission:**
   ```bash
   cargo run --example commission_ip --features driver,tracing -- \
     --manual <code-from-step-1> \
     --paa-dir <chip>/credentials/production/paa-root-certs \
     --cd-root <chip>/credentials/production/cd-certs \
     --out fabric.json -vv
   ```
   On success it prints `✅ commissioned` with the fabric id, compressed fabric id,
   assigned node id, and peer public key, and writes `fabric.json`.
4. **Record the device** in `docs/tested-devices.md`.

### Troubleshooting (Path A)

| Symptom | Cause / fix |
|---|---|
| Attestation chain failure | The PAA snapshot may predate this vendor's PAA approval. Refresh the connectedhomeip snapshot, or pull the vendor's PAA from the DCL web UI (`https://webui.dcl.csa-iot.org`). This is the cached-snapshot limitation, not a bug. |
| `device discovery failed: ... not found via mDNS` **with a manual code** | Manual pairing codes carry only the **4-bit short discriminator**, but `resolve_commissionable` matches the device's full 12-bit `D` record exactly, so mDNS discovery cannot match. Use the **QR code** (`--qr`, full discriminator) or bypass discovery with `--addr` (see below). Tracked as a known limitation (short-discriminator matching). |
| Device not found via mDNS (QR) | Busy LAN / mDNS noise. Pass `--addr` to dial directly. |
| `No route to host (os error 65)` on **macOS** at the first send | macOS **Local Network privacy** (Sequoia+) is blocking the binary. `tccutil` cannot reset Local Network — use System Settings → Privacy & Security → **Local Network** and enable the terminal app actually hosting the run (e.g. iTerm, not Terminal), or run the binary on Linux (no such gate). Confirm with a raw `python3` UDP send: if Python reaches the device but the binary does not, it is the privacy gate, not the code. |
| Window closed mid-run | Commissioning windows are time-limited (~15 min). Re-open it from the existing ecosystem and retry. |

**Finding the device address for `--addr`.** Browse mDNS, then resolve the host:
```bash
dns-sd -Z _matterc._udp local.        # find the instance whose D's upper nibble matches your short discriminator
dns-sd -Gv4v6 <HOST>.local.           # resolve to addresses
```
`--addr` accepts an IPv4 (`192.168.1.248:5540`), a global/ULA IPv6
(`[fdxx::1]:5540`), or a **link-local** IPv6 with a numeric zone id
(`[fe80::...%11]:5540`, where `11` is the interface index from
`python3 -c "import socket; print(socket.if_nametoindex('en0'))"`). The example
binds an IPv4 socket automatically when `--addr` is IPv4.

## Path B — rs-matter test device (reproducible; no production material)

1. Bring up `project-chip/rs-matter`'s on-network example device on Linux/Pi (or
   connectedhomeip `chip-lighting-app` as an alternative). Record its QR/manual code.
2. Commission with the **bundled CSA test roots** (the `⚠ TEST ATTESTATION ROOTS`
   banner will print — expected here):
   ```bash
   cargo run --example commission_ip --features driver -- --qr "MT:..."
   ```
   This validates the device's *test* attestation chain against the bundled test roots
   and needs no DCL material.

## Cross-verification against matter.js (required)

Per CLAUDE.md: "if our output differs from matter.js, we are wrong by default."
Tooling: `cargo xtask trace-diff` (see `docs/m6-cross-verification.md` for the
schema, comparison rules, and the report from the P110M run).

1. **matter.js run.** Open a commissioning window (Path A step 1), then:
   ```bash
   cd xtask/scripts/capture-commission-trace && npm install   # once
   node index.js --manual <code> --out ../../../runs/matterjs-p110m.jsonl
   ```
   The script removes its own fabric from the device before exiting
   (best-effort; if decommissioning fails it still exits 0 — clear the
   slot from the owning ecosystem's app if needed).
2. **Rust run.** Re-open the window, then (`mkdir -p runs` first if running
   this step before step 1 — the example does not create the directory):
   ```bash
   cargo run --example commission_ip --features driver,tracing,wiretrace -- \
     --manual <code> \
     --paa-dir <chip>/credentials/production/paa-root-certs \
     --cd-root <chip>/credentials/production/cd-certs \
     --trace-out runs/rust-p110m.jsonl -vv
   ```
3. **Diff.**
   ```bash
   cargo xtask trace-diff runs/rust-p110m.jsonl runs/matterjs-p110m.jsonl
   ```
4. **Triage every DIVERGENT / DECODE-FAIL** (we are wrong by default):
   - our encoding bug → fix the crate, re-run both captures;
   - legitimate variance → add a rule to `xtask/src/trace_diff.rs::rules()`
     with a comment explaining WHY;
   - matter.js quirk → document in `docs/matter-js-comparison.md`.
5. Record the outcome in `docs/m6-cross-verification.md` and
   `docs/tested-devices.md`.

`runs/` is gitignored — the decrypted traces contain the device's DAC
chain, our issued NOC/IPK, and fabric identifiers. Never commit them.

### Troubleshooting (cross-verification)

| Symptom | Cause / fix |
|---|---|
| `Cannot find module '@matter/general'` | Run `npm install` inside `xtask/scripts/capture-commission-trace/` first — it is not bundled. |
| matter.js still holds the fabric slot after a failed run | `removeNode` is best-effort (exits 0 on failure). Clear the slot from the owning ecosystem app (e.g. Apple Home → Remove from Home for our added fabric). |
| QR path (`--qr`) not finding the device | `--qr` is implemented and typings-verified but was not device-tested during the P110M run. Prefer `--manual` for both sides if you have the manual code — the discovery method does not affect trace content. Fall back to `--addr` if mDNS fails regardless. |
| matter.js mDNS with `--manual` discovers slowly | matter.js uses the 4-bit short discriminator from the manual code — this is native and works, but may take a moment if several devices are advertising. |
| Rust `--manual` does not discover (short discriminator) | Existing limitation: see the Path A troubleshooting table (`--addr` workaround). Using `--qr` for the Rust run and `--manual` for matter.js is fine — trace content is discovery-method-independent. |
| Trace file is 0 bytes | The run never reached an encrypted exchange (discovery or PASE setup failed before any message was recorded). Check STDERR for the failure; this is not a capture-tooling bug. |
| `sequence diverges at aligned index N` / `trace lengths differ within the compared range` | **Expected on the first real run, not a tooling bug.** The differ requires both dialogues to walk the same message sequence, but matter.js issues its own IM read/write pattern inside the window (it reads more attributes than our minimal driver). Read the ±3-message context block the differ prints: if the extra/missing messages are matter.js-only IM traffic, that is sequence-level triage — record it in the report and assess whether our driver should send the same reads. A mismatch in the PASE/CASE handshake or a StatusReport body, by contrast, is "wrong by default" — investigate our side first. |

## Reference-implementation mapping

How `commission_ip` maps onto the two reference implementations (lifted from the M6.6
design §8):

| Our piece | connectedhomeip | matter.js |
|---|---|---|
| `Commissioner` cursor (policy) | `AutoCommissioner` | `ControllerCommissioner` |
| `commission()` driver (IO per stage) | `DeviceCommissioner::PerformCommissioningStep` | `MessageExchange` / `ExchangeManager` |
| `im` module | `CommandSender` / `ReadClient` | `InteractionClient` |
| PASE / CASE bridges | `PASESession` / `CASESession` | `PaseClient` / `CaseClient` |
| `AsyncDatagram` + mDNS | `Transport::Base` + `Dnssd::ResolverProxy` | `UdpChannel` / mDNS scanner |
