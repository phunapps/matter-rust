# Runbook: first IP commission with `commission_ip` (M6.6)

This is an **operator-touch** procedure, not a CI gate. It commissions a real,
IP-reachable Matter device end to end with the pure-Rust `commission_ip` example,
and cross-verifies the on-wire trace against matter.js.

> A factory-fresh Matter device is **BLE-only** until it has network credentials,
> and BLE commissioning transport is deferred post-v1.0. So the IP path below uses
> a device that is **already on the network** and put into a commissioning window.
> The factory-fresh **Tuya-plug-over-BLE** run named in CLAUDE.md's M6 validation
> remains **deferred until BLE lands** — it is not dropped, just not yet runnable.

## Path A — real LAN device via open commissioning window (production)

1. **Open a commissioning window.** Pick a device already commissioned to another
   ecosystem (Apple Home / Google Home / SmartThings). In that app, choose
   "pair to another app" / "add to another service". It opens a commissioning
   window (Enhanced Commissioning Method) and shows an **11-digit manual code** or
   a QR code. Note: this is a **second-fabric** commission — our controller adds its
   own fabric to the device. Ongoing multi-admin *management* (cross-fabric ACLs,
   fabric removal) stays deferred per CLAUDE.md; only the commissioning act is in scope.
2. **Get production attestation roots.** Clone `project-chip/connectedhomeip` (or
   sparse-checkout `credentials/production/`):
   - `--paa-dir` → `credentials/production/paa-root-certs/` (the example loads every
     `*.der` there).
   - `--cd-root` → the CSA CD signing cert under `credentials/production/cd-certs/`
     (PEM). Pin the exact filename when you run it.
3. **Commission:**
   ```bash
   cargo run --example commission_ip --features driver,tracing -- \
     --manual <code-from-step-1> \
     --paa-dir <chip>/credentials/production/paa-root-certs \
     --cd-root <chip>/credentials/production/cd-certs/<csa-cd>.pem \
     --out fabric.json -vv
   ```
   On success it prints `✅ commissioned` with the fabric id, compressed fabric id,
   assigned node id, and peer public key, and writes `fabric.json`.
4. **Record the device** in `docs/tested-devices.md`.

### Troubleshooting (Path A)

| Symptom | Cause / fix |
|---|---|
| Attestation chain failure | The PAA snapshot may predate this vendor's PAA approval. Refresh the connectedhomeip snapshot, or pull the vendor's PAA from the DCL web UI (`https://webui.dcl.csa-iot.org`). This is the cached-snapshot limitation, not a bug. |
| Device not found via mDNS | Busy LAN / mDNS noise. Pass `--addr <device-ip>:5540` to dial directly (find the IP from your router or the device's existing app). |
| Window closed mid-run | Commissioning windows are time-limited (~15 min). Re-open it from the existing ecosystem and retry. |

## Path B — rs-matter test device (reproducible; no production material)

1. Bring up `project-chip/rs-matter`'s on-network example device on Linux/Pi (or
   connectedhomeip `chip-lighting-app` as an alternative). Record its QR/manual code.
2. Commission with the **bundled CSA test roots** (the `⚠ TEST ATTESTATION ROOTS`
   banner will print — expected here):
   ```bash
   cargo run --example commission_ip --features driver -- --qr "MT:..."
   ```
   This validates the device's *test* attestation chain against the bundled test roots
   and needs no DCL material.

## Cross-verification against matter.js (required)

Per CLAUDE.md: "if our output differs from matter.js, we are wrong by default."

1. Commission the **same** device with matter.js (`@matter/main` controller example),
   with verbose logging, and capture its on-wire messages.
2. Capture our trace with `--features driver,tracing -vv` (per-step spans:
   discover / pase / case / exchange, field names aligned best-effort to matter.js keys).
3. Diff message-by-message: PASE handshake (PBKDFParamReq/Resp, Pake1/2/3), the IM
   invoke/read sequence (ArmFailSafe, attestation, CSR, AddTrustedRoot, AddNOC,
   network, CommissioningComplete), and the CASE SIGMA-1/2/3 exchange. Investigate any
   byte-level divergence before declaring success.

## Reference-implementation mapping

How `commission_ip` maps onto the two reference implementations (lifted from the M6.6
design §8):

| Our piece | connectedhomeip | matter.js |
|---|---|---|
| `Commissioner` cursor (policy) | `AutoCommissioner` | `ControllerCommissioner` |
| `commission()` driver (IO per stage) | `DeviceCommissioner::PerformCommissioningStep` | `MessageExchange` / `ExchangeManager` |
| `im` module | `CommandSender` / `ReadClient` | `InteractionClient` |
| PASE / CASE bridges | `PASESession` / `CASESession` | `PaseClient` / `CaseClient` |
| `AsyncDatagram` + mDNS | `Transport::Base` + `Dnssd::ResolverProxy` | `UdpChannel` / mDNS scanner |
