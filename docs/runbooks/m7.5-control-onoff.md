# Runbook: control a device with `control_onoff` (M7.5)

This is an **operator-touch** procedure, not a CI gate. It commissions a real,
IP-reachable Matter device and then drives the generated `matter-clusters`
codecs over a **fresh operational CASE session** — reading and toggling
`OnOff`, and writing/reading `BasicInformation.NodeLabel` — and cross-verifies
the on-wire dialogue against matter.js.

It builds on M6.6. For commissioning prerequisites (opening a commissioning
window, production attestation roots, `--addr`/mDNS troubleshooting, the macOS
Local Network privacy gate), see [`m6.6-first-commission.md`](m6.6-first-commission.md).
Everything there about getting `commission_ip` to reach the device applies
verbatim to `control_onoff` — it shares the same CLI surface.

> The example commissions **and** operates in one process: it mints an
> ephemeral fabric in memory, commissions the device, then opens a second,
> fresh operational session and runs the five operations. Persistent fabric
> identity and reconnect-later are M8 (`matter-controller`).

## 1. Run the Rust example (with trace capture)

Use the test attestation roots for a CSA test device, or `--paa-dir`/`--cd-root`
for a production device (see the M6.6 runbook). With wire-trace capture on:

```bash
cargo run --example control_onoff --features driver,tracing,wiretrace -- \
  --manual <code> \
  --trace-out runs/rust-onoff.jsonl -vv
```

Expected stdout (values depend on the device's starting state):

```text
setup payload: vid=... pid=... discriminator=... passcode=<redacted>
commissioning… (PASE → attestation → NOC → CASE → CommissioningComplete)
✅ commissioned: peer_node_id=0x0000000000000002
opening operational CASE session…
   operational session established (local id N)
   read   OnOff.OnOff            = false
   invoke OnOff.Toggle           -> OK
   read   OnOff.OnOff            = true (was false)
   write  BasicInformation.NodeLabel = "matter-rust"
   read   BasicInformation.NodeLabel = "matter-rust"
✅ control_onoff complete
```

The example exits non-zero if any operation fails, if `OnOff` does not change
across the Toggle, or if `NodeLabel` does not echo back the written value.

## 2. Run the matter.js oracle

Factory-reset the device first so it is commissionable again (each commission
consumes a fabric slot — see Caveats). Then:

```bash
cd xtask/scripts/capture-onoff-trace
npm install          # first time only; pins @matter 0.17.1 (same as the sibling)
node index.js --manual <code> --out ../../../runs/matterjs-onoff.jsonl
cd -
```

The script commissions, connects, runs the same five operations, removes its
own fabric, and writes the trace. Progress goes to STDERR; the trace is written
line-by-line so a crash still leaves a partial trace on disk.

## 3. Cross-verify

```bash
cargo xtask trace-diff runs/rust-onoff.jsonl runs/matterjs-onoff.jsonl
```

Read the per-message verdict table. **Success = exit 0** with every aligned
message `MATCH` or `MATCH*` and **0 DIVERGENT**.

> **What this checks — and what it does not.** `trace-diff` cuts its comparison
> window at `CommissioningComplete` (M6 design), so it verifies the
> **commissioning** flow byte-for-byte; it does **not** compare the operational
> OnOff/NodeLabel ops. Those are not a clean byte-diff target anyway: matter.js's
> high-level controller does batched multi-attribute reads across several CASE
> sessions and emits **no `WriteRequest`** for the label, whereas this example
> issues a tight 5-message sequence with an explicit write. The operational layer
> is instead validated by (a) the device accepting and acting on our exact bytes
> (the example asserts the decoded values — OnOff flips, NodeLabel echoes) and
> (b) codec + IM-framing byte-parity unit tests. See
> [`../m7.5-control-onoff-verification.md`](../m7.5-control-onoff-verification.md).

Expect some `TheirsOnly` rows: matter.js issues extra structure-discovery reads
and removes its fabric (`RemoveFabric`) at the end. Both are **reported, not
failures** — exactly as in the M6 commissioning cross-check. `trace-diff` does
decode IM `WriteRequest`/`WriteResponse` (0x06/0x07) and aligns them on their
`(cluster, attribute)` target, for any write that falls inside a compared window.

### Troubleshooting (cross-verification)

| Symptom | Cause / fix |
|---|---|
| `DIVERGENT` on an `OnOff` read | The two runs started from different `OnOff` states. The diff aligns by target, not value; a differing decrypted value on a matched read is a real divergence — re-run both from a factory-reset device so they start identically. |
| `DIVERGENT` on the `WriteRequest` | Check both sides write `BasicInformation.NodeLabel` (ep 0, 0x0028/0x0005) with the same `"matter-rust"` string. |
| matter.js run floods the trace with reports | Confirm `autoSubscribe: false` in the script's controller options (a subscribe-all storm would add many unmatched reports). |
| Operational `not found via mDNS` | The device's operational `_matter._tcp` record had not propagated yet. Re-run; the resolve loop polls for a few seconds. |

## Caveats

- **Fabric slots.** `control_onoff` does **not** decommission the device on exit
  (matching `commission_ip`); each run consumes one of the device's fabric slots
  (a typical plug holds a handful). **Factory-reset** the device to reclaim them.
  Decommissioning from Rust (`OperationalCredentials.RemoveFabric`) is M8 work.
- **Test roots banner.** With no `--paa-dir`/`--cd-root`, the run trusts CSA
  *test* PAA/CD roots and prints a warning. Not valid for production trust
  decisions.
- **Decrypted traces** contain DAC chains, NOCs, and fabric ids — keep them
  under `runs/` (gitignored). Do not commit captured traces.

## Reference-implementation mapping

- `matter-interaction` builds the IM `ReadRequest` / `InvokeRequest` /
  `WriteRequest` envelopes; `matter-clusters` (generated) encodes/decodes the
  cluster payloads; `matter-commissioning::driver` (`run_case`,
  `secured_round_trip`) carries them over the operational CASE session.
- The matter.js side uses the high-level `CommissioningController` →
  `PairedNode` → cluster-client API (`getOnOffAttribute`, `toggle`,
  `setNodeLabelAttribute`).
